# Анализ диаграммы классов

Замечания:
1. В языке UML для обозначения отношений между классами предусмотрены разные виды стрелок. Здесь же вместо них
везде добавлены одинаковые стрелочки с подписями, при этом острой необходимости в этом не видно. Например, есть стрелочки с подписью
`assosiation`: зачем, когда в UML есть специальная стрелка для этого типа связи?
2. Использование разного нейминга:
    - `available` (в классе `Worker`) и `isOccupied` (в классе `RepairPlace`)
    - `repairAssignedTo` — можно было использовать вместо этого `assignee`, чтобы было существительное, как и в других местах
    - `numberOfPlaces: int`, но при этом `freePlaces: int` в `RepairZone`
3. Не хватает указаний кратности
4. Tight coupling: на диаграмме очень много двунаправленных связей. Например, у каждого объекта класса `Car` есть `productionPlan`, при этом у каждого
`ProductionPlan` есть `List<Car>`.
5. У `Repair` указано одновременно и поле `repairAssignedTo`, и отношение `assignedWorker`
6. `numberOfPlaces` в `RepairZone` не нужно, когда можно посчитать количество из `List<RepairPlace>`
7. Используется очень много примитивных типов, где можно использовать `enum` или объекты сложных типов. Например, `productionStage` и `role`
следует сделать `enum`.

# Описание mad diagram
Диаграмма выглядит довольно сложно и запутано. Было бы значительно более читаемо, если бы она была разделена на несколько по модулям и пакетам

Видим по диаграмме несколько пакетов:
- CLI
- Validation (для CLI)
- Configuration
- Factory
- Experiment
- Generator
- Worker
- Report
- Persistence (присутствует одновременно в модулях Criterion и Experiment)
- TestStatistics
- GoodnessOfFitTest
- CVCalculator
- Test

Связи очень сложно читать из-за визуальной перегруженности, но всё же можно понять, что эта система позволяет выполнять расчётные эксперименты 
трёх видов: CriticalValue, Power, TimeComplexity. Каждая из этих фабрик порождает эксперимент, в котором описаны шаги для исполнение Worker'ом.
У экспериментов отдельным шагом идёт генерация отчёта. Результаты эксперимента сохраняются в постоянное хранилище. Worker'ы вычисляют тестовые
статистики для проверки соответствия статистической модели данным, которые порождаются генератором и сохраняются в соответствующее хранилище
тестовых данных.

Общее назначение системы — быстрая проверка статистических гипотез.
